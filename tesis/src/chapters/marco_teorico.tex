\section{SISTEMAS DISTRIBUIDOS}

Los sistemas distribuidos constituyen uno de los pilares fundamentales del desarrollo de software moderno. El crecimiento exponencial de usuarios, dispositivos conectados y servicios digitales ha generado la necesidad de construir aplicaciones capaces de operar sobre múltiples servidores y ubicaciones geográficas de manera simultánea~\parencite{tanenbaum2017}. Como resultado, las organizaciones han evolucionado desde aplicaciones monolíticas ejecutadas en una única máquina hacia arquitecturas distribuidas capaces de ofrecer mayor disponibilidad, escalabilidad y tolerancia a fallos.

Un sistema distribuido puede definirse como un conjunto de componentes de software que se ejecutan en diferentes nodos computacionales y que se comunican mediante una red para alcanzar un objetivo común~\parencite{coulouris2012}. Aunque físicamente los componentes se encuentran separados, desde la perspectiva del usuario final el sistema debe comportarse como una única entidad lógica.

La Figura~\ref{fig:sistema-distribuido} muestra un ejemplo simplificado de un sistema distribuido basado en microservicios, donde diferentes servicios colaboran para atender solicitudes de los usuarios.

\subsection{Definición de Sistema Distribuido}

\textcite{coulouris2012} definen un sistema distribuido como ``una colección de computadoras independientes que para los usuarios se presenta como un sistema único y coherente''. \textcite{tanenbaum2017} complementan esta definición señalando que un sistema distribuido es ``una colección de elementos de computación autónomos que aparecen ante sus usuarios como un solo sistema coherente''.

Estas definiciones destacan dos características fundamentales. La primera es la independencia física de los componentes, ya que cada nodo posee sus propios recursos de procesamiento, memoria y almacenamiento. La segunda es la transparencia, es decir, la capacidad de ocultar la complejidad interna del sistema para que los usuarios perciban una única aplicación integrada.

Los sistemas distribuidos se encuentran presentes en numerosos escenarios tecnológicos actuales, incluyendo plataformas de comercio electrónico, sistemas bancarios, redes sociales, servicios de streaming y soluciones basadas en la nube.

\subsection{Características de los Sistemas Distribuidos}

Los sistemas distribuidos poseen características particulares que los diferencian de las aplicaciones tradicionales ejecutadas en una sola máquina~\parencite{coulouris2012}:

\begin{itemize}
    \item \textbf{Concurrencia}: múltiples componentes ejecutan operaciones simultáneamente sobre recursos compartidos.
    \item \textbf{Escalabilidad}: capacidad de incrementar recursos mediante la incorporación de nuevos nodos (horizontal) o el aumento de capacidad de los existentes (vertical).
    \item \textbf{Transparencia}: oculta la complejidad interna del sistema al usuario final.
    \item \textbf{Tolerancia a fallos}: capacidad de continuar funcionando incluso cuando algunos componentes presentan fallos.
    \item \textbf{Heterogeneidad}: posibilidad de integrar componentes desarrollados con diferentes tecnologías y plataformas.
\end{itemize}

\subsection{Desafíos de los Sistemas Distribuidos}

A pesar de sus ventajas, los sistemas distribuidos presentan desafíos importantes que incrementan significativamente la complejidad de diseño e implementación~\parencite{kleppmann2017}:

\begin{itemize}
    \item Coordinación entre componentes que se ejecutan en diferentes nodos.
    \item Retrasos de red, pérdida de paquetes y particiones.
    \item Sincronización de información entre diferentes bases de datos.
    \item Gestión de fallos parciales (un servicio falla mientras otros continúan).
    \item Pérdida o duplicación de mensajes entre servicios.
\end{itemize}

Estos problemas han dado origen a conceptos como consistencia eventual, transacciones distribuidas y patrones como \textit{Transactional Outbox}, los cuales serán analizados en secciones posteriores.

En el contexto de la presente investigación, resulta especialmente relevante el problema de la pérdida o duplicación de eventos durante la comunicación entre servicios distribuidos, lo cual motiva el desarrollo de la librería propuesta.

\section{ARQUITECTURA DE MICROSERVICIOS}

La arquitectura de microservicios representa uno de los enfoques más adoptados para el desarrollo de aplicaciones distribuidas modernas. Su principal objetivo es dividir sistemas complejos en componentes pequeños, independientes y especializados, permitiendo una mayor flexibilidad en el desarrollo, despliegue y mantenimiento~\parencite{newman2015}.

A diferencia de los sistemas monolíticos tradicionales, donde toda la lógica de negocio se encuentra agrupada en una única aplicación, los microservicios distribuyen las funcionalidades en múltiples servicios autónomos que colaboran entre sí mediante mecanismos de comunicación bien definidos~\parencite{richardson2018}.

\subsection{Concepto de Microservicios}

\textcite{newman2015} define los microservicios como ``servicios pequeños y autónomos que trabajan juntos''. Cada servicio implementa una capacidad específica del negocio, posee su propia lógica, almacenamiento de datos y ciclo de vida de despliegue.

Un principio fundamental de esta arquitectura es que cada microservicio debe ser responsable de una única funcionalidad del dominio, promoviendo así una alta cohesión y un bajo acoplamiento entre componentes~\parencite{fowler2014}.

\section{COMUNICACIÓN ENTRE MICROSERVICIOS}

La comunicación entre microservicios constituye uno de los elementos fundamentales de las arquitecturas distribuidas modernas. Debido a que cada microservicio opera de manera independiente, es necesario establecer mecanismos que permitan el intercambio de información y la coordinación de procesos entre los distintos servicios~\parencite{richardson2018}.

La selección del modelo de comunicación impacta directamente en aspectos como el rendimiento, la escalabilidad, la disponibilidad, la resiliencia y el grado de acoplamiento entre los servicios.

\subsection{Comunicación Síncrona}

La comunicación síncrona se caracteriza porque un servicio realiza una solicitud a otro servicio y permanece esperando una respuesta antes de continuar con la ejecución de su proceso. Las tecnologías más utilizadas son REST (HTTP/JSON), gRPC y GraphQL~\parencite{richardson2018}.

Este modelo resulta apropiado cuando se requiere una respuesta inmediata. Sin embargo, introduce acoplamiento temporal entre servicios y puede generar fallos en cascada cuando un servicio no se encuentra disponible.

\subsection{Comunicación Asíncrona}

La comunicación asíncrona permite que un servicio envíe información sin necesidad de esperar una respuesta inmediata. El servicio productor genera un mensaje o evento que es enviado a un sistema intermediario conocido como \textit{broker de mensajería}. Posteriormente, uno o varios consumidores procesan dicho mensaje cuando se encuentran disponibles~\parencite{kleppmann2017}.

Este modelo reduce significativamente el acoplamiento entre componentes y permite que los servicios evolucionen y escalen de forma independiente. Las tecnologías más utilizadas incluyen Apache Kafka, RabbitMQ y Amazon SQS.

\section{ARQUITECTURA ORIENTADA A EVENTOS}

La arquitectura orientada a eventos, conocida también como \textit{Event-Driven Architecture} (EDA), es un estilo arquitectónico en el que los componentes del sistema se comunican mediante la producción, publicación, consumo y procesamiento de eventos~\parencite{richardson2018}.

Un evento representa un hecho significativo que ha ocurrido dentro del sistema. A diferencia de una solicitud directa, un evento no expresa una orden hacia otro servicio, sino que informa que algo ya sucedió. Por ejemplo, la creación de una orden (\textit{OrderCreated}), la confirmación de un pago (\textit{PaymentConfirmed}) o la actualización de inventario (\textit{StockReserved}).

Este enfoque permite que los servicios se mantengan desacoplados, ya que el productor del evento no necesita conocer directamente a los consumidores. La Figura~\ref{fig:arquitectura_eventos} muestra una representación general de una arquitectura orientada a eventos.

\input{diagrams/arquitectura_eventos}

En este modelo intervienen tres elementos principales:

\begin{itemize}
    \item \textbf{Productor de eventos}: genera el evento cuando ocurre un cambio importante en el sistema.
    \item \textbf{Broker de eventos}: recibe, almacena y distribuye los eventos.
    \item \textbf{Consumidor de eventos}: procesa el evento de acuerdo con su funcionalidad de negocio.
\end{itemize}

Una de las principales ventajas de EDA es la extensibilidad: nuevos consumidores pueden suscribirse a eventos existentes sin modificar el servicio productor. Sin embargo, también introduce desafíos como la gestión del orden de los eventos, el manejo de duplicados y la necesidad de garantizar que los eventos no se pierdan antes de ser publicados~\parencite{kleppmann2017}.

Este tipo de arquitectura es especialmente relevante para la presente investigación, ya que el patrón \textit{Transactional Outbox} busca resolver uno de los principales problemas asociados a la publicación de eventos: garantizar que un evento generado por una operación de negocio sea almacenado y publicado de forma confiable.

\section{SISTEMAS DE MENSAJERÍA Y BROKERS DE EVENTOS}

Los sistemas de mensajería constituyen uno de los componentes fundamentales de las arquitecturas distribuidas modernas. Su principal objetivo es permitir el intercambio de información entre servicios de manera desacoplada, confiable y escalable.

Dentro de una arquitectura orientada a eventos, la comunicación entre productores y consumidores se realiza mediante un intermediario denominado \textit{broker de eventos}. La Figura~\ref{fig:broker_eventos} muestra el papel que desempeña un broker dentro de esta arquitectura.

Kafka organiza la información en estructuras denominadas \textit{topics}, dentro de las cuales los eventos son almacenados de forma secuencial y persistente. \subsection{RabbitMQ}

RabbitMQ es un broker de mensajería basado en el protocolo AMQP (\textit{Advanced Message Queuing Protocol}), ampliamente utilizado para implementar sistemas distribuidos desacoplados.

A diferencia de Kafka, RabbitMQ se basa principalmente en colas de mensajes. Los productores envían mensajes a un componente denominado \textit{exchange}, el cual se encarga de enrutar dichos mensajes hacia una o varias colas según reglas de configuración específicas.

\subsection{Importancia de los Brokers en la Entrega Confiable de Eventos}

A pesar de las capacidades que ofrecen Kafka y RabbitMQ, la existencia de un broker no garantiza por sí sola que un evento sea publicado correctamente. Cuando una aplicación debe almacenar información en una base de datos y simultáneamente publicar un evento en el broker, si una de estas operaciones falla, el sistema puede entrar en un estado inconsistente.

Esta situación es conocida como el problema de la doble escritura (\textit{Dual Write Problem}) y será desarrollada en la siguiente sección.

\section{CONSISTENCIA DE DATOS EN SISTEMAS DISTRIBUIDOS}

La consistencia de datos es uno de los principales desafíos en los sistemas distribuidos. En una aplicación monolítica, las operaciones se ejecutan dentro de una única base de datos mediante transacciones ACID. En una arquitectura distribuida, los datos pueden estar repartidos entre múltiples servicios y bases de datos independientes~\parencite{kleppmann2017}.

\subsection{Consistencia Fuerte y Consistencia Eventual}

La \textbf{consistencia fuerte} establece que todos los componentes del sistema observan el mismo estado de los datos inmediatamente después de que una operación se confirma. Este modelo es común en sistemas centralizados con transacciones ACID.

La \textbf{consistencia eventual} es un modelo en el cual los datos pueden no estar sincronizados inmediatamente después de una operación, pero alcanzarán un estado consistente después de cierto tiempo~\parencite{vogels2009}. Este enfoque es frecuente en arquitecturas distribuidas modernas porque permite priorizar la disponibilidad y la escalabilidad.

En una arquitectura basada en eventos, un servicio modifica su estado local y posteriormente publica un evento para que otros servicios actualicen su propia información. Durante el intervalo entre la modificación y el procesamiento del evento por parte de los demás servicios, puede existir una inconsistencia temporal que se resuelve eventualmente.

\subsection{Teorema CAP}

El teorema CAP, formulado por \textcite{brewer2012}, establece que un sistema distribuido no puede garantizar simultáneamente las tres propiedades siguientes:

\begin{itemize}
    \item \textbf{Consistencia (C)}: todos los nodos observan el mismo valor de los datos al mismo tiempo.
    \item \textbf{Disponibilidad (A)}: cada solicitud recibe una respuesta, aunque algunos nodos puedan estar fallando.
    \item \textbf{Tolerancia a particiones (P)}: el sistema continúa funcionando incluso ante fallos de comunicación entre nodos.
\end{itemize}

En presencia de una partición de red, el sistema debe elegir entre mantener la consistencia inmediata o preservar la disponibilidad. Por esta razón, muchas arquitecturas de microservicios adoptan modelos de consistencia eventual combinados con mecanismos de compensación y patrones como \textit{Transactional Outbox}.

\subsection{Transacciones Distribuidas y sus Limitaciones}

Una transacción distribuida es una operación que involucra múltiples recursos o bases de datos independientes. El protocolo clásico \textit{Two-Phase Commit} (2PC) coordina a los participantes en dos fases: preparación y confirmación~\parencite{tanenbaum2017}.

Aunque 2PC puede garantizar atomicidad, presenta limitaciones importantes en sistemas modernos:

\begin{itemize}
    \item Bloqueo de recursos durante la fase de preparación.
    \item Dependencia de un coordinador central (punto único de falla).
    \item Reducción significativa de disponibilidad y rendimiento.
    \item Incompatibilidad con brokers de mensajería que no soportan el protocolo XA.
\end{itemize}

Por estas razones, en arquitecturas de microservicios se prefieren mecanismos basados en consistencia eventual y eventos, como el patrón \textit{Transactional Outbox}.

\section{PROBLEMA DE LA DOBLE ESCRITURA}

El problema de la doble escritura, conocido también como \textit{Dual Write Problem}, es uno de los principales desafíos en arquitecturas de microservicios orientadas a eventos~\parencite{richardson2018}. Este problema aparece cuando un servicio debe realizar dos operaciones relacionadas sobre sistemas diferentes: una escritura en una base de datos y una publicación de eventos en un broker de mensajería.

El problema surge porque ambas operaciones no forman parte de una misma transacción atómica. La base de datos y el broker de eventos son sistemas independientes, por lo tanto, no existe una garantía directa de que ambas operaciones se completen correctamente.

\input{diagrams/doble_escritura}

Como se observa en la figura, el microservicio confirma correctamente la operación en la base de datos, pero falla al publicar el evento en el broker. En este caso, el dato existe en la base de datos, pero los demás servicios nunca reciben la notificación correspondiente.

\subsection{Escenarios de Inconsistencia}

Los escenarios de inconsistencia que pueden producirse son:

\begin{enumerate}
    \item \textbf{Datos guardados, evento no publicado}: la base de datos confirma la operación, pero el broker no recibe el evento debido a una falla de red o caída del servicio.
    \item \textbf{Evento publicado, datos no confirmados}: el evento se publica exitosamente, pero la transacción de base de datos falla y se revierte.
    \item \textbf{Evento duplicado}: ante reintentos sin control, el mismo evento se publica múltiples veces.
    \item \textbf{Broker no disponible}: la aplicación completa la operación de base de datos pero no puede conectarse al broker.
\end{enumerate}

\subsection{Impacto en los Microservicios}

El impacto del problema de la doble escritura puede ser considerable en sistemas donde múltiples servicios dependen de eventos para coordinar procesos de negocio. Las consecuencias incluyen: pérdida de eventos críticos, procesamiento duplicado de operaciones, desincronización entre servicios y dificultad para recuperar el estado correcto del sistema.

\subsection{Necesidad de un Mecanismo Confiable}

Para resolver este problema es necesario utilizar un mecanismo que permita coordinar la persistencia de datos y la publicación de eventos sin depender de una transacción distribuida entre la base de datos y el broker.

El patrón \textit{Transactional Outbox} responde precisamente a esta necesidad: almacena los eventos en una tabla especial dentro de la misma base de datos del servicio, utilizando la misma transacción que guarda los datos de negocio. Posteriormente, un proceso asíncrono se encarga de publicar los eventos pendientes hacia el broker.

\section{PATRÓN TRANSACTIONAL OUTBOX}

El patrón \textit{Transactional Outbox} es una solución arquitectónica ampliamente utilizada en microservicios para garantizar la entrega confiable de eventos sin emplear transacciones distribuidas~\parencite{richardson2018}. Fue formalizado por Chris Richardson como parte de su catálogo de patrones para microservicios.

La idea principal consiste en almacenar los eventos de integración dentro de una tabla especial denominada \textit{Outbox}, utilizando la misma transacción de base de datos que persiste los datos de negocio. De esta manera, si la transacción se confirma exitosamente, tanto los datos como el evento quedan almacenados de forma atómica.

Posteriormente, un componente independiente se encarga de leer periódicamente los registros pendientes de la tabla Outbox y publicarlos en el broker de mensajería correspondiente.

La Figura~\ref{fig:outbox_general} muestra la arquitectura general del patrón.

\subsection{Funcionamiento del Patrón}

El funcionamiento del patrón puede dividirse en dos fases principales:

\begin{enumerate}
    \item \textbf{Fase transaccional}: el servicio almacena los datos de negocio y el evento asociado dentro de la misma transacción local. Esto garantiza atomicidad: ambas operaciones se confirman o se revierten conjuntamente.
    \item \textbf{Fase de publicación}: un proceso independiente (Polling Publisher) consulta periódicamente la tabla Outbox, identifica eventos pendientes, los publica en el broker y los marca como procesados.
\end{enumerate}

\subsection{Estrategias de Publicación}

Existen dos estrategias principales para la publicación de eventos desde la tabla Outbox:

\begin{itemize}
    \item \textbf{Polling Publisher}: un proceso programado consulta periódicamente la tabla Outbox para identificar eventos pendientes. Es la estrategia más simple de implementar y no requiere infraestructura adicional.
    \item \textbf{Transaction Log Tailing}: una herramienta de CDC (como Debezium) monitorea el log transaccional de la base de datos y detecta automáticamente las inserciones en la tabla Outbox.
\end{itemize}

La presente investigación implementará la estrategia de \textit{Polling Publisher} por su simplicidad y menor dependencia de infraestructura externa.

\subsection{Ventajas del Patrón}

El patrón Transactional Outbox ofrece múltiples beneficios~\parencite{richardson2018}:

\begin{itemize}
    \item Garantiza que los eventos no se pierdan al almacenarlos transaccionalmente.
    \item Elimina el problema de la doble escritura.
    \item Evita la necesidad de transacciones distribuidas (2PC).
    \item Permite recuperación ante fallos temporales del broker.
    \item Mantiene independencia entre el servicio productor y los consumidores.
    \item Es compatible con cualquier broker de mensajería.
\end{itemize}

\subsection{Limitaciones del Patrón}

Aunque Transactional Outbox proporciona una solución robusta, también introduce ciertos desafíos:

\begin{itemize}
    \item La tabla Outbox requiere mecanismos de limpieza para evitar crecimiento excesivo.
    \item La publicación no es inmediata sino diferida (latencia entre escritura y publicación).
    \item No evita completamente la posibilidad de eventos duplicados (\textit{at-least-once delivery}).
    \item Los consumidores deben diseñarse con idempotencia para manejar duplicados.
\end{itemize}

\section{IDEMPOTENCIA Y ENTREGA CONFIABLE DE EVENTOS}

La entrega confiable de eventos es un aspecto fundamental en arquitecturas orientadas a eventos. En entornos distribuidos no siempre es posible garantizar que un mensaje sea entregado exactamente una sola vez~\parencite{kleppmann2017}. Fallos de red, reinicios de servicios o mecanismos de reintento pueden provocar entregas duplicadas.

\subsection{Modelos de Entrega de Mensajes}

Los sistemas de mensajería trabajan bajo diferentes modelos de entrega, cada uno con garantías distintas. La Tabla~\ref{tab:delivery_models} resume estos modelos.

\begin{table}[H]
    \centering
    \caption{Modelos de entrega de mensajes en sistemas distribuidos.}
    \begin{tabular}{|l|p{4.5cm}|p{4.5cm}|}
        \hline
        \textbf{Modelo} & \textbf{Garantía} & \textbf{Riesgo} \\
        \hline
        At-most-once & El mensaje se entrega como máximo una vez. & Pérdida de mensajes. \\
        \hline
        At-least-once & El mensaje se entrega al menos una vez. & Mensajes duplicados. \\
        \hline
        Exactly-once & El mensaje se procesa exactamente una vez. & Complejidad extrema de implementación. \\
        \hline
    \end{tabular}
    \label{tab:delivery_models}
\end{table}

En la práctica, la mayoría de los sistemas distribuidos implementan \textit{at-least-once delivery} combinado con consumidores idempotentes, ya que \textit{exactly-once} es extremadamente difícil de garantizar en sistemas distribuidos~\parencite{kleppmann2017}.

\subsection{Concepto de Idempotencia}

La idempotencia es una propiedad mediante la cual una operación puede ejecutarse varias veces con los mismos datos y producir siempre el mismo resultado final. En el contexto de eventos, un consumidor idempotente puede recibir el mismo evento más de una vez sin generar efectos duplicados en el sistema.

Por ejemplo, si un consumidor recibe dos veces el evento \textit{PaymentConfirmed}, no debería registrar dos pagos distintos. En cambio, debe detectar que el evento ya fue procesado y evitar repetir la operación.

\subsection{Patrón Idempotent Consumer}

El patrón \textit{Idempotent Consumer} propone que cada evento tenga un identificador único~\parencite{richardson2018}. Antes de procesar un evento, el consumidor verifica si dicho identificador ya fue registrado como procesado:

\begin{enumerate}
    \item Se recibe el evento con su identificador único.
    \item Se consulta el registro de eventos procesados.
    \item Si el evento no ha sido procesado, se ejecuta la lógica de negocio y se almacena el identificador.
    \item Si el evento ya fue procesado, se descarta sin ejecutar acciones adicionales.
\end{enumerate}

\subsection{Relación con Transactional Outbox}

El patrón \textit{Transactional Outbox} garantiza que los eventos sean registrados de manera persistente (evitando pérdida), pero dependiendo del mecanismo de publicación, un evento podría ser enviado más de una vez. Por esta razón, Transactional Outbox se combina con consumidores idempotentes: Outbox evita la pérdida de eventos y la idempotencia evita que los duplicados produzcan inconsistencias.

La combinación de ambos enfoques permite construir sistemas más confiables en arquitecturas distribuidas donde los fallos temporales son inevitables.

\section{CHANGE DATA CAPTURE (CDC)}

Change Data Capture (CDC) es una técnica que permite identificar, capturar y transmitir cambios realizados sobre los datos almacenados en una base de datos, convirtiéndolos en eventos que pueden ser consumidos por otros sistemas~\parencite{kleppmann2017}.

\subsection{Concepto y Métodos de Implementación}

A diferencia de los mecanismos tradicionales basados en consultas periódicas, CDC permite reaccionar a los cambios casi en tiempo real. La Figura~\ref{fig:cdc_general} muestra la arquitectura general de CDC.

\input{diagrams/cdc_general}

Existen tres métodos principales para implementar CDC:

\begin{itemize}
    \item \textbf{Triggers}: ejecutan lógica automáticamente ante operaciones INSERT, UPDATE o DELETE. Afectan el rendimiento de la base de datos.
    \item \textbf{Polling}: consulta periódica a la tabla para detectar cambios. Simple pero introduce latencia.
    \item \textbf{Transaction Log Tailing}: lectura directa del log transaccional (WAL en PostgreSQL, Binlog en MySQL). Método preferido por su bajo impacto y mínima latencia.
\end{itemize}

\subsection{Debezium como Herramienta de CDC}

Debezium es una plataforma de código abierto desarrollada por Red Hat para implementar CDC sobre diferentes motores de bases de datos~\parencite{debezium2020}. Su funcionamiento se basa en la lectura de los transaction logs y la generación automática de eventos publicados en Apache Kafka.

\subsection{CDC y Transactional Outbox}

CDC suele utilizarse como complemento del patrón Transactional Outbox. La aplicación almacena los eventos en la tabla Outbox dentro de la transacción, y una herramienta CDC detecta automáticamente los nuevos registros y los publica hacia el broker. Esta combinación elimina la necesidad de un proceso de polling, pero requiere infraestructura adicional (Debezium, Kafka Connect).

La presente investigación implementa la estrategia de polling como mecanismo principal de publicación, manteniendo CDC como alternativa documentada para escenarios de mayor escala.

\section{TECNOLOGÍAS DE IMPLEMENTACIÓN}

En esta sección se describen las tecnologías seleccionadas para la implementación de la librería propuesta.

\subsection{Java}

Java es un lenguaje de programación orientado a objetos, fuertemente tipado y de propósito general, desarrollado originalmente por Sun Microsystems en 1995 y actualmente mantenido por Oracle Corporation~\parencite{bloch2018}.

Entre las características relevantes de Java para la presente investigación se encuentran:

\begin{itemize}
    \item Independencia de plataforma mediante la Java Virtual Machine (JVM).
    \item Gestión automática de memoria (Garbage Collection).
    \item Ecosistema maduro de librerías y frameworks.
    \item Soporte robusto para concurrencia y programación multihilo.
    \item Amplia adopción en el desarrollo de aplicaciones empresariales y microservicios.
\end{itemize}

\subsection{Spring Boot}

Spring Boot es un framework de desarrollo de aplicaciones Java que simplifica la configuración y el despliegue de aplicaciones basadas en el ecosistema Spring~\parencite{walls2019}. Proporciona configuración automática, servidores embebidos y convenciones que reducen significativamente el código necesario para crear aplicaciones productivas.

Las características relevantes de Spring Boot para esta investigación incluyen:

\begin{itemize}
    \item \textbf{Auto-configuración}: detecta las dependencias del proyecto y configura automáticamente los componentes necesarios.
    \item \textbf{Spring Data JPA}: facilita el acceso a bases de datos relacionales mediante repositorios declarativos.
    \item \textbf{Spring Scheduling}: permite la ejecución de tareas programadas (utilizado para el polling de la tabla Outbox).
    \item \textbf{Spring Kafka}: proporciona integración nativa con Apache Kafka para la producción y consumo de eventos.
    \item \textbf{Gestión transaccional}: mediante la anotación \texttt{@Transactional}, permite demarcar transacciones de forma declarativa.
    \item \textbf{Starters}: módulos preconfigurados que simplifican la incorporación de funcionalidades.
\end{itemize}

\subsection{PostgreSQL}

PostgreSQL es un sistema gestor de bases de datos relacional de código abierto, reconocido por su robustez, extensibilidad y cumplimiento de estándares SQL~\parencite{kleppmann2017}. Soporta tipos de datos avanzados como JSONB (utilizado para almacenar el payload de los eventos en la tabla Outbox) y ofrece un log transaccional (Write-Ahead Log) compatible con herramientas de CDC.

\subsection{Apache Kafka}

Apache Kafka es la plataforma de streaming de eventos seleccionada como broker principal para la librería~\parencite{narkhede2017}. Sus características de persistencia, replicación y procesamiento distribuido lo convierten en la opción más adoptada para arquitecturas de microservicios de gran escala.

\subsection{Docker y Testcontainers}

Docker permite la creación de contenedores que encapsulan aplicaciones y sus dependencias en entornos aislados y reproducibles. Testcontainers es una librería Java que permite ejecutar contenedores Docker durante las pruebas de integración, proporcionando instancias reales de bases de datos y brokers para validar el comportamiento de la librería en condiciones cercanas a producción.
